# 2026-04-27 路由契约对齐设计

## 1. 背景与问题

当前后端入口路由与契约文档存在偏差：

- `backend/cmd/server/main.go` 使用模块前缀（如 `/intake`、`/workbench`、`/render`）
- `docs/01-product/api-conventions.md` 与 `docs/modules/*/contract.md` 定义为资源路径（如 `/projects`、`/assets`、`/drafts`、`/tasks`）

该偏差会导致新功能继续按错误前缀开发，造成代码与契约持续漂移。

## 2. 目标

- 将入口路由装配与模块契约完全对齐。
- 统一约束：对外 API 以资源路径为准，不以模块名为准。
- 允许单模块注册多个资源根路径（例如 A 与 E 模块）。
- 本轮不保留旧前缀兼容层。

## 3. 设计约束

- 只调整路由装配和文档口径，不改变业务语义。
- 响应结构、错误码、处理流程保持不变。
- 旧路径不重定向，直接下线。

## 4. 方案对比

### 方案 A（采用）

`main.go` 仅保留 `/api/v1` 分组，将该分组对象传给各模块。模块在各自 `routes.go` 内注册资源子路径。

优点：
- 与 `contract.md` 天然一致。
- 支持一个模块多个资源根路径。
- 入口稳定、职责清晰。

缺点：
- 各模块路由定义会更完整，需要约束模块内自行维护资源前缀。

### 方案 B（未采用）

`main.go` 继续做模块二级分组，但将前缀改成“主资源”。

问题：
- A/E 存在多资源根，分组抽象不自然。
- 仍然容易在入口层堆积契约细节。

### 方案 C（未采用）

所有资源前缀集中在 `main.go` 统一声明，再分发给模块函数。

问题：
- `main.go` 易膨胀。
- 模块边界变弱，后续维护成本上升。

## 5. 最终设计

### 5.1 路由装配

- `main.go` 仅负责：
  - 初始化中间件
  - 创建 `/api/v1` 基础分组
  - 调用模块 `RegisterRoutes(v1, db)`
- A-E 模块各自在内部注册契约资源路径。

### 5.2 资源路径口径

以各模块 `contract.md` 为准：

- A：`/projects`、`/assets` 及其子路径
- B：`/parsing/*`
- C：`/ai/*`
- D：`/drafts/{draft_id}`
- E：`/drafts/{draft_id}/*` 与 `/tasks/{task_id}`

### 5.3 文档一致性

更新 `CLAUDE.md`：

- 路由映射示例改为资源路径示例
- 明确写出“以 `docs/modules/*/contract.md` 为唯一契约来源”
- 移除模块前缀示例（`/intake`、`/workbench`、`/render`）

## 6. 数据流与错误处理

- 请求处理链路保持：Client -> Gin Router -> Handler -> Service -> DB
- 本轮只改“路径匹配入口”，不改业务处理和响应结构
- 未命中路径保持 404 默认行为
- 错误码范围与语义保持模块契约定义

## 7. 验收标准

- 入口不再使用模块前缀分组
- `CLAUDE.md` 与 `contract.md` 路径口径一致
- 关键路径抽样可通过：`/projects`、`/assets`、`/parsing`、`/ai`、`/drafts`、`/tasks`
- 旧路径请求返回 404（不兼容）

## 8. 风险与缓解

风险：调用方若仍使用旧前缀将返回 404。

缓解：
- 在 `CLAUDE.md` 和本设计文档中显式声明旧前缀下线
- 后续由实施计划补充契约守护测试，防止再次漂移

## 9. 非目标

- 本轮不实现兼容路由
- 本轮不做 handler/service 重构
- 本轮不引入新的认证与权限逻辑

## 10. 决策结论

采用方案 A：入口只保留 `/api/v1`，模块自行注册资源子路径；文档全面对齐 `contract.md`。
